# Q&A — Concept Recap

These are real questions asked during learning sessions, answered in the student's own words, with corrections and clarifications added. This is more valuable than any guide — it captures the exact points of confusion and how they were resolved.

---

## Q1 — Why never use root credentials? What's the difference between a user and a role?

**Answer given:**
Root creds are dangerous because they have unrestricted access — a wrong command could delete everything without warning. Instead you create an IAM user with only the permissions it needs. A user can have multiple roles in its permission stack. A role is like a decision-making scenario but can't execute on its own — it needs a user.

**Correction:**
The root credentials reasoning is right. But user vs role needs sharpening:

- **User** = a human identity with long-lived credentials (access key + secret). You log in as a user. `smartdocs-dev` is your user.
- **Role** = an identity for *AWS services*, not humans. Lambda, EC2, S3 assume roles at runtime to get temporary permissions. No access key, no password — just "I am Lambda, I need S3 access."

A role doesn't live inside a user's permission stack. They're separate concepts. Users are for humans, roles are for services. Lambda can't have an access key — it assumes a role instead. `smartdocs-lambda-role` is the role Lambda assumes when it runs.

## Q2 — What is a presigned URL and why does it exist?

**Answer given:**
A presigned URL is pre-generated by a Lambda function. User clicks upload → Lambda generates the URL → URL has upload capability → file is uploaded (like a GDrive link). `aws s3 cp` uploads directly but requires AWS credentials — you can't share those with a client-side user, so presigned URLs are the practical solution.

**Correction:**
Spot on. The key insight — "credentials cannot be shared to the client-side" — is exactly right. A presigned URL is a temporary, cryptographically signed URL that carries the permission baked in. It expires after N seconds and requires zero AWS credentials from the client. The GDrive analogy holds well.

`aws s3 cp` = for developers with credentials (you, in your terminal).
Presigned URL = for end users with no AWS credentials (anyone with a browser).

## Q3 — What are the three things every Lambda function has? Cold start vs warm start?

**Answer given:**
1. An event — char count, word count, doc_id generator (pre-defined tasks executed on signal)
2. Handler — handles the event, the if/else block returning 200 or 500
3. Context — metadata about execution, like a barcode generator for each event

**Correction:**
The three things are:
1. **Handler** — the entry point AWS calls (`lambda_handler`)
2. **Trigger** — what invokes it (API Gateway, S3, schedule, etc.)
3. **Role** — what AWS services it's allowed to touch

`event` and `context` are the two *parameters* the handler receives, not the three things a Lambda has. `event` = the input data (HTTP body or S3 notification). `context` = read-only runtime metadata (timeout remaining, function name) — not a barcode generator, just info about the current execution.

**Cold start vs warm start — correction needed:**
Cold/warm start has nothing to do with trigger type. It's about container lifecycle:

- **Cold start** — first invocation (or after idle period). AWS spins up a fresh container, loads your code, initialises the runtime. Adds 200-600ms.
- **Warm start** — subsequent invocations reuse the existing container. Just your code runs, no spin-up cost.

Applies equally to API Gateway and S3 triggers. Think of it like a car engine — cold start is turning the key on a winter morning, warm start is driving again 5 minutes after parking.

## Q4 — What is AWS_PROXY integration?

**Answer given:**
AWS_PROXY is like a refiner/converter that cleans up raw data into your format. Use it when you want full control in code, not API Gateway mapping templates.

**Correction:**
The "full control in code" part is right, but AWS_PROXY is actually the *opposite* of a refiner. It means API Gateway passes the request through **as-is** — no transformation, no filtering. The entire raw HTTP request lands in your Lambda's `event` object exactly as it came in.

Without AWS_PROXY, API Gateway would transform the request using mapping templates before passing it to Lambda. AWS_PROXY = "don't touch it, just pass it straight through." Your Lambda gets the full raw request and is responsible for parsing it — which is why you do `json.loads(event.get("body", "{}"))`.

**Bonus — what does the raw event look like?**
```json
{
  "routeKey": "POST /documents",
  "headers": { "content-type": "application/json" },
  "body": "{\"s3_key\": \"documents/test.txt\", \"count_words\": true}",
  "isBase64Encoded": false
}
```
`body` is a string, not a dict — that's why `json.loads()` is needed. `event.get("body", "{}")` safely defaults to an empty JSON string if no body was sent.

**Where does `"count_words": true` come from?**
From the client — whoever makes the request. In testing that's you via curl. In production it's JavaScript's `fetch()` in a browser. `curl` and `fetch` produce identical HTTP POST requests. The client controls which action runs by what it puts in the body.

## Q5 — scan vs get_item in DynamoDB

**Answer given:**
Scan loops over all available body content — all objects and values. Fine at small scale but expensive when there are thousands of items. Always prefer `get_item` which fetches only one item by its key.

**Correction:**
Right understanding, one precision fix: `scan` loops over every **item (row) in the DynamoDB table**, not "body content" (that's HTTP terminology). At 10 items it's fine. At 10 million items it reads all 10 million even if you only wanted one.

`get_item` goes directly to one item using the partition key — like knowing exactly which shelf in a warehouse to go to. No looping.

Rule: prefer `get_item` when you know the `doc_id`. Use `scan` only when you genuinely need everything, and always add a `Limit`.

There's also a third option — `query`. Faster than scan, more flexible than get_item. Used when you have a sort key and want multiple related items for one partition. Coming in a future challenge.

## Q6 — S3 event trigger vs API Gateway event structure

**Answer given:**
S3 triggers have `Records` which is an array of records `[0][1][2]...`. API Gateway uses raw HTTP JSON format. Extraction:
```python
bucket = record["s3"]["bucket"]["name"]
key = record["s3"]["object"]["key"]
```

**Correction:**
Spot on. One small fix — API Gateway doesn't send "raw response body from DynamoDB". It sends a raw HTTP request from the *client* (browser/curl). DynamoDB isn't involved at that point. The difference is:

- S3 event → `event["Records"][0]["s3"]`
- API Gateway event → `event["body"]` (JSON string from the client)

`Records` is a list because S3 can batch multiple file events into one notification. Always index with `[0]` in practice.

## Q7 — Why keep S3 trigger and API Gateway handler as separate Lambda functions?

**Answer given:**
Both have different response structures. API Gateway uses raw HTTP JSON via curl/fetch. S3 triggers use `Records` array. Mixing them could cause the function to misinterpret between `record["s3"]["bucket"]["name"]` and `body.get("s3_key")` — fine at small scale but risky at enterprise scale.

**Correct — plus three more reasons:**

1. **Different event structures** — you'd need `if "Records" in event` checks everywhere, adding complexity and misrouting risk. Exactly as described.

2. **Different response requirements** — API Gateway Lambda *must* return `{"statusCode": 200, "body": "..."}`. S3 trigger Lambda returns nothing (`None`). Mixing forces conditional return formats.

3. **Different IAM needs** — S3 trigger only needs S3 read + DynamoDB write. HTTP handler needs S3 read/write + DynamoDB read/write + presigned URL generation. Separate functions = separate roles = least privilege. Tighter security.

4. **Operational clarity** — when something breaks, you check one function's CloudWatch logs with a clear purpose. A combined function makes debugging harder — you'd have to figure out which path failed.

Rule of thumb: one Lambda function = one responsibility = one trigger type.
